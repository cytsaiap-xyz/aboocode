## 1. 什麼是 Harness Engineering

Harness Engineering 是圍繞大語言模型（LLM）構建的**運行時治理架構**。它不是模型本身的能力，而是一套外部系統，負責：

- **管理上下文**：控制模型看到什麼、何時壓縮、何時注入
- **治理工具**：決定哪些工具可用、何時授權、如何記錄
- **協調 Agent**：管理多個 Agent 的生命週期、通訊與隔離
- **持久化知識**：跨會話保存記憶、偏好與專案知識
- **保障安全**：權限檢查、工作區隔離、失敗恢復

```
┌─────────────────────────────────────────────┐
│                   使用者                      │
├─────────────────────────────────────────────┤
│              Harness（治理層）                │
│  ┌─────────┐ ┌──────────┐ ┌──────────────┐  │
│  │ 上下文  │ │ 工具治理 │ │  Agent 排程  │  │
│  │ 管理器  │ │   管線   │ │    管理器    │  │
│  └─────────┘ └──────────┘ └──────────────┘  │
├─────────────────────────────────────────────┤
│                 LLM（模型層）                 │
└─────────────────────────────────────────────┘
```

**關鍵區分**：模型提供推理能力；Harness 提供運行時治理。缺少 Harness 的 Agent 只是一個有工具的聊天機器人。

## 2. 為什麼需要 Harness Engineering

### 2.1 模型的邊界

即使是最強大的 LLM，也存在以下根本限制：

| 限制 | 說明 |
|------|------|
| 無持久記憶 | 每次會話都從零開始，不記得上次討論的內容 |
| 無環境感知 | 不知道當前工作目錄、Git 分支、檔案系統狀態 |
| 無權限概念 | 無法區分唯讀操作和破壞性操作 |
| 上下文有限 | 即使 200K token 視窗，複雜專案也會溢出 |
| 無自我驗證 | 無法可靠地驗證自己的輸出正確性 |
| 無失敗恢復 | API 超時、token 溢出時沒有自動恢復機制 |

### 2.2 簡單 Agent 包裝的不足

最常見的「Agent 框架」只是在模型外面加一個工具呼叫迴圈：

```
while not done:
    response = llm.chat(messages)
    if response.has_tool_calls:
        results = execute_tools(response.tool_calls)
        messages.append(results)
```

這種方式存在嚴重問題：

| 問題 | 後果 |
|------|------|
| 上下文膨脹 | 對話越長 token 越多，最終溢出或品質下降 |
| 無治理 | 任何工具隨時可呼叫，沒有權限控制 |
| 知識丟失 | 會話結束後所有知識消失 |
| 無可觀測性 | 不知道模型做了什麼、為什麼做 |
| 單點故障 | API 錯誤直接終止整個流程 |
| 無隔離 | 所有操作共享同一個工作區，互相干擾 |

### 2.3 解決方案 — 13 個 Harness Phase

| Phase | 子系統 | 解決的問題 |
|-------|--------|------------|
| 01 | 三層上下文壓縮 | 上下文膨脹 |
| 02 | 系統提示詞動態邊界 | Token 浪費、快取失效 |
| 03 | 會話全文持久化 | 壓縮後原始資料丟失 |
| 04 | 壓縮後身份重注入 | 壓縮後角色遺忘 |
| 05 | 原生記憶系統 | 跨會話知識丟失 |
| 06 | 流式工具執行器 | 工具並發安全 |
| 07 | 增強 Hook 系統 | 可擴展性不足 |
| 08 | 延遲工具載入 | 工具數量膨脹 |
| 09 | 後台 Agent 執行 | 任務阻塞 |
| 10 | 工具治理管線 | 無治理、無可觀測性 |
| 11 | 獨立驗證 Agent | 無法自我驗證 |
| 12 | 工作區隔離模式 | 操作互相干擾 |
| 13 | 失敗恢復管線 | 單點故障 |

## 3. 架構分層

```
┌─────────────────────────────────────────────────────────┐
│  使用者互動層（User Interaction Layer）                    │
│  CLI / TUI (Ink) / API                                  │
├─────────────────────────────────────────────────────────┤
│  會話管理層（Session Management Layer）                    │
│  上下文壓縮 │ 提示詞邊界 │ 身份重注入 │ 全文持久化        │
├─────────────────────────────────────────────────────────┤
│  工具執行層（Tool Execution Layer）                        │
│  流式執行器 │ 治理管線 │ Hook 系統 │ 延遲載入             │
├─────────────────────────────────────────────────────────┤
│  Agent 排程層（Agent Scheduling Layer）                    │
│  後台 Agent │ 驗證 Agent │ 工作區隔離                     │
├─────────────────────────────────────────────────────────┤
│  持久化層（Persistence Layer）                             │
│  記憶系統 │ Transcript 存檔 │ 設定檔管理                  │
├─────────────────────────────────────────────────────────┤
│  擴展層（Extension Layer）                                │
│  Plugin 架構 │ MCP 伺服器 │ 自訂工具                     │
└─────────────────────────────────────────────────────────┘
```

**資料流向**：使用者輸入 → 動態提示詞組裝 → 上下文壓縮檢查 → LLM 呼叫 → 工具治理管線 → 結果注入 → 回應輸出

| 流式工具執行 | 有 | 有 | 基本一致 |

| 獨立驗證 | 無 | 有 | Aboocode 新增唯讀驗證 Agent |

## 產品經理視角總結

Aboocode 不是在模型上面包一層殼，而是一套**完整的運行時治理架構**。

把 LLM 想像成一顆強大的 CPU：它能做複雜的計算，但沒有作業系統，它什麼都做不了。作業系統管理記憶體、排程程序、控制權限、處理故障——這正是 Harness Engineering 對 LLM 做的事情。

13 個 Phase 不是功能的堆疊，而是一個完整系統的不同面向。它們共同回答一個問題：**如何讓一個無狀態的語言模型，安全、可靠、可觀測地在真實環境中持續工作？**
